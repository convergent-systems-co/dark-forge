# project.yaml Configuration Guide

Complete reference for the `project.yaml` configuration file used by consuming repositories.

## Overview

`project.yaml` is the project-level configuration file that customizes governance behavior for your repository. It lives at the **project root** (not inside `.ai/`) and is auto-generated by the Code Manager during the first `/startup` session if it doesn't already exist.

You can also create it manually by copying a language template:

```bash
cp .ai/governance/templates/python/project.yaml project.yaml
```

Available templates: `python/`, `node/`, `react/`, `go/`, `csharp/`, `bicep/`, `terraform/`.

## Configuration Lookup Flow

Settings are resolved using a two-level lookup:

1. **`project.yaml`** (project root) — project-specific overrides
2. **`config.yaml`** (inside `.ai/`) — platform defaults

**Merge rules:**
- Scalar values (strings, numbers, booleans) in `project.yaml` **override** `config.yaml` defaults
- Object values are **deep-merged** — project-level keys are applied on top of defaults
- Array values in `project.yaml` **replace** the default arrays entirely

## Schema Validation

`project.yaml` is validated against [`governance/schemas/project.schema.json`](../../governance/schemas/project.schema.json). The schema enforces `additionalProperties: false` at every level, so unrecognized keys will fail validation.

## Top-Level Keys

| Key | Type | Required | Default | Description |
|-----|------|----------|---------|-------------|
| `name` | string | Yes | — | Project name identifier |
| `language` | string | Yes | — | Primary programming language (`python`, `typescript`, `go`, `java`) |
| `framework` | string \| null | No | `null` | Framework used (`react`, `django`, `spring-boot`, `fastapi`, `express`, etc.) |

```yaml
name: my-app
language: typescript
framework: react
```

## governance

Controls governance pipeline behavior.

| Key | Type | Default | Description |
|-----|------|---------|-------------|
| `policy_profile` | string | `"default"` | Policy profile to use. Maps to a file in `governance/policy/`. Options: `default`, `fin_pii_high`, `infrastructure_critical`, `reduced_touchpoint` |
| `parallel_coders` | integer | `5` | Maximum concurrent Coder agents during parallel dispatch. Range: `-1` to `10`. Set to `-1` for unlimited (context-monitored) |
| `skip_panel_validation` | boolean | `false` | Opt out of mandatory panel validation. Use with caution — CI will not block on missing panels |
| `ai_submodule_pin` | string \| null | `null` | Pin the `.ai` submodule to a specific commit SHA or tag (e.g., `"v2.1.0"`) |

```yaml
governance:
  policy_profile: default
  parallel_coders: 5
  skip_panel_validation: false
  ai_submodule_pin: null
```

### Policy Profiles

| Profile | Use Case | Auto-Merge | Key Behavior |
|---------|----------|------------|--------------|
| `default` | Standard internal applications | Enabled (with conditions) | Balanced automation and oversight |
| `fin_pii_high` | Financial/PII — SOC2, PCI-DSS, HIPAA, GDPR | Disabled | 3-approver override required |
| `infrastructure_critical` | Production infrastructure, deployment configs | Conditional | Mandatory architecture and SRE review |
| `reduced_touchpoint` | Near-full autonomy (Phase 5e) | Enabled | Human approval only for policy overrides and dismissed security-critical findings |

## conventions

Project coding conventions that inform AI agents and review panels.

### conventions.testing

| Key | Type | Default | Description |
|-----|------|---------|-------------|
| `framework` | string | — | Test framework (`pytest`, `jest`, `vitest`, `junit`, `go test`) |
| `coverage_target` | integer | — | Minimum test coverage percentage (0-100) |
| `naming` | string | — | Test naming convention (`test_`, `Test`, `spec.`) |

### conventions.style

| Key | Type | Default | Description |
|-----|------|---------|-------------|
| `max_line_length` | integer | — | Maximum line length |
| `indent` | string | — | Indentation type: `spaces` or `tabs` |
| `indent_size` | integer | — | Number of spaces per indent level |

### conventions.git

| Key | Type | Default | Description |
|-----|------|---------|-------------|
| `branch_pattern` | string | — | Branch naming pattern (e.g., `itsfwcp/{type}/{number}/{name}`) |
| `commit_style` | string | — | Commit message style: `conventional` or `freeform` |
| `pr_template` | boolean | — | Whether to use PR templates |

```yaml
conventions:
  testing:
    framework: pytest
    coverage_target: 80
    naming: test_
  style:
    max_line_length: 100
    indent: spaces
    indent_size: 4
  git:
    branch_pattern: "itsfwcp/{type}/{number}/{name}"
    commit_style: conventional
    pr_template: true
```

## repository

GitHub repository settings applied by `init.sh` via the GitHub API. All fields are optional — defaults come from `config.yaml`.

| Key | Type | Default | Description |
|-----|------|---------|-------------|
| `auto_merge` | boolean | `false` | Enable GitHub auto-merge. Required for the agentic loop to merge PRs after CI + approval |
| `delete_branch_on_merge` | boolean | `true` | Automatically delete feature branches after PR merge |
| `allow_squash_merge` | boolean | `true` | Allow squash-and-merge strategy |
| `allow_merge_commit` | boolean | `true` | Allow merge commit strategy |
| `allow_rebase_merge` | boolean | `true` | Allow rebase-and-merge strategy |

### repository.codeowners

| Key | Type | Default | Description |
|-----|------|---------|-------------|
| `enabled` | boolean | `true` | Generate a CODEOWNERS file during init |
| `default_owner` | string | — | Default code owner for all files (e.g., `@SET-Apps/approvers`) |
| `rules` | array | — | Ordered CODEOWNERS rules (later rules take precedence) |

Each rule in `rules`:

| Key | Type | Description |
|-----|------|-------------|
| `pattern` | string | File pattern (CODEOWNERS glob syntax, e.g., `/.github/workflows/`) |
| `owners` | array of strings | GitHub teams or users who own matching files |

### repository.branch_protection

| Key | Type | Description |
|-----|------|-------------|
| `expected_rulesets` | array | Rulesets expected to be configured. Validated (not applied) by init.sh |

Each ruleset:

| Key | Type | Required | Description |
|-----|------|----------|-------------|
| `name` | string | Yes | Name of the expected ruleset |
| `required_approving_review_count` | integer | No | Minimum approving reviews |
| `require_code_owner_review` | boolean | No | Whether code owner review is required |

```yaml
repository:
  auto_merge: true
  delete_branch_on_merge: true
  codeowners:
    enabled: true
    default_owner: "@my-org/team"
    rules:
      - pattern: "/.github/workflows/"
        owners: ["@my-org/devops"]
      - pattern: "/infra/"
        owners: ["@my-org/platform"]
  branch_protection:
    expected_rulesets:
      - name: "Pull Request - Reviewer"
        required_approving_review_count: 1
        require_code_owner_review: true
```

## personas

Array of persona file paths to activate. Paths are relative to `governance/personas/`.

```yaml
personas:
  - agentic/devops-engineer.md
  - agentic/code-manager.md
  - agentic/coder.md
  - agentic/iac-engineer.md
  - agentic/tester.md
  - agentic/project-manager.md
```

## panels

Array of panel file paths to activate. Paths are relative to `governance/prompts/reviews/`.

```yaml
panels:
  - code-review.md
  - security-review.md
  - threat-modeling.md
  - cost-analysis.md
  - documentation-review.md
  - data-governance-review.md
```

## instructions

Freeform project-specific instructions appended to `CLAUDE.md`. Use multiline YAML syntax.

```yaml
instructions: |
  This is a Django REST API. Always use class-based views.
  Database migrations must be reviewed by the platform team.
  Never modify the authentication middleware directly.
```

## escalation

Cross-repo issue escalation configuration. When enabled, the agentic loop detects framework-level issues and escalates them to the upstream governance repository.

| Key | Type | Default | Description |
|-----|------|---------|-------------|
| `enabled` | boolean | `false` | Enable cross-repo escalation (must be explicitly set to `true`) |
| `target_repository` | string | `"SET-Apps/ai-submodule"` | Upstream repository for escalations (format: `owner/repo`) |
| `auth_method` | string | `"GITHUB_TOKEN"` | Authentication method: `GITHUB_TOKEN`, `PAT`, or `GITHUB_APP` |
| `token_secret` | string | — | Repository secret name for PAT (only when `auth_method: PAT`) |
| `app_id_secret` | string | — | Secret name for GitHub App ID (only when `auth_method: GITHUB_APP`) |
| `private_key_secret` | string | — | Secret name for GitHub App private key (only when `auth_method: GITHUB_APP`) |
| `rate_limit` | integer | `3` | Maximum escalations per agentic session (range: 1-10) |

### escalation.criteria

Which detection criteria trigger escalations. All default to `true` when escalation is enabled.

| Key | Type | Default | Description |
|-----|------|---------|-------------|
| `path_based` | boolean | `true` | Escalate when issues involve `.ai/governance/` files |
| `workflow_failure` | boolean | `true` | Escalate when governance workflow fails due to framework code errors |
| `policy_gap` | boolean | `true` | Escalate when the policy engine encounters unmatched scenarios |
| `schema_validation` | boolean | `true` | Escalate when governance schema validation fails on valid-looking artifacts |
| `init_failure` | boolean | `true` | Escalate when init.sh fails due to missing governance files |

```yaml
escalation:
  enabled: true
  target_repository: SET-Apps/ai-submodule
  auth_method: GITHUB_TOKEN
  rate_limit: 3
  criteria:
    path_based: true
    workflow_failure: true
    policy_gap: true
    schema_validation: true
    init_failure: true
```

## Complete Examples

### Python (FastAPI)

```yaml
name: my-fastapi-app
language: python
framework: fastapi

governance:
  policy_profile: default
  parallel_coders: 5

conventions:
  testing:
    framework: pytest
    coverage_target: 80
    naming: test_
  style:
    max_line_length: 100
    indent: spaces
    indent_size: 4
  git:
    branch_pattern: "itsfwcp/{type}/{number}/{name}"
    commit_style: conventional

repository:
  auto_merge: true
  delete_branch_on_merge: true
```
